Now let's proceed to the definition and coding of our tools. These tools will be Python functions, decorated with the @tool decorator of the LangGraph framework. This way, the various assistants will be able to access them in an agentic way, meaning that the assistatns will call these tools as they see fit, and they will handle the tool outputs appropriately. 


Recall that we have three assistants in our architecture. Each one will have access to its own set of tools, as shown below: 



Based on our previous discussion about system prompts, we made clear that the Spending and the Recommendation assistant have access to some tools. So let's start by coding the tools of these two.


Spending assistant tools

The code below implements the fetch_spending_events tool. 


import sqlite3
from langchain_core.tools import tool


@tool
def fetch_spending_events(sql_query: str) -> str:
    """
    Fetches the spending events for a given user from an SQL database.
    The SQL table is called "spending_events". The table schema is given below:
            {
				event_id: INTEGER PRIMARY KEY AUTOINCREMENT,
				customer_id: INTEGER,
				plan_name: TEXT NOT NULL,
				billing_start: DATE NOT NULL,
				billing_end: DATE NOT NULL,
				amount_due: REAL
            }
    In our case, customer_id is always 1, so you can simply ignore this field.
    """

    conn = sqlite3.connect("demo.db")
    cursor = conn.cursor()

    cursor.execute(sql_query)
    result = cursor.fetchall()

    message = ""
    for entry in result:
        message += f"\n{entry}"

    conn.close()

    return message


The fetch_spending_events tool accepts an SQL query as a string argument, it establishes a connection to the demo database, then executes the SQL query upon the database, and it finally returns the result of the query in a string format. However, it is crucial to understand how this works under the hood: 

  • Tool argument: the sql_query argument will be dynamically created by the LLM that powers the Spending assistant. That means that we will not have a programmatic control over this argument.

  • Tool docstring: the docstring of the tool is of paramount importance. Effectively, the assistant will evaluate this docstring to check if the tool is right for its task and hence decide if it needs to call it or not. Further, the docstring should entail important information about the scope of the tool. For example, here we specify that the assistant should look at the spending_events SQL table and its schema. Without this information, we cannot expect the LLM to form an appropriate SQL query for the tool argument. In case we observe frequent wrong tool argument passing, we can experiment with different docstrings to see if they help.

Recommendation assistant tools

The code below implements the list_supported_plans and fetch_plan_description tools. 


@tool
def list_supported_plans() -> list[str]:
    """Returns a list of plans the assistant has knowledge about."""
    message = ["Standard Plan", "Eco Plan", "Night Plan"]

    return f"Supported plans are: {message}"


@tool
def fetch_plan_description(sql_query: str) -> str:
    """
    Fetches the detailed description of an electricity plan for an SQL database.
    The SQL table is called "electricity_plans". The table schema is given below:
            {
                plan_id: INTEGER PRIMARY KEY AUTOINCREMENT,
                plan_name: TEXT NOT NULL,
                plan_description: TEXT NOT NULL,
                selling_points: TEXT NOT NULL
            }
    """

    conn = sqlite3.connect("demo.db")
    cursor = conn.cursor()

    cursor.execute(sql_query)
    charging_events = cursor.fetchall()

    message = ""
    for event in charging_events:
        message += f"\n{event}"

    # Close the connection
    conn.close()

    return message


The  list_supported_plans tool takes no arguments and simply returns a list of electricity plans that are offered by the electricity company. We expect this tool to be used if the user asks something along the lines of "What electricity plans do you offer?" or if the user specifies a plan that does not exist and we have to re-direct them to only supported plans. Notice here the docstring, which is a concise message that specifies the purpose of the tool.

The fetch_plan_description tool is very similar to the fetch_spending_events tool. It connects to the demo database and fetches a more detailed description of a particular plan. The corresponding LLM that fuels the Recommendation assistant will have to generate the SQL query that fetches information about the plan of interest.


Common tools

Our assistants can have some tools in common. Here, we define the CompleteOrEscalate tool, this time as a BaseModel class instead of a function. This tool is adopted from the LangGraph tutorial that we follow (https://langchain-ai.github.io/langgraph/tutorials/customer-support/customer-support/#assistants).  


from pydantic import BaseModel


class CompleteOrEscalate(BaseModel):
    """A tool to mark the current task as completed and/or to escalate control of the dialog to the primary assistant, who can re-route the dialog based on the user's needs."""

    cancel: bool = True
    reason: str

    class Config:
        json_schema_extra = {
            "example": {
                "cancel": True,
                "reason": "User changed their mind about the current task.",
            },
            "example 2": {
                "cancel": True,
                "reason": "I have fully completed the task.",
            },
            "example 3": {
                "cancel": False,
                "reason": "I can try searching into another database.",
            },
        }


The CompleteOrEscalate tool has a very special purpose: to enable the conversation to escape from a particular specialized assistant/workflow. This hints another dimension of tools; they can be used for routing purposes as well. For example, assume that a user interacts with the Spending assistant and at some point of the conversation they ask "Tell me about the Standard plan". Apparently, this question does not fall under the scope of the Spending assistant, rather it is more appropriate for the Recommendation assistant. The Spending assistant can then trigger the CompleteOrEscalate tool to delegate back to the Primary assistant, who will hopefully then steer the flow towards the Recommendation assistant.

In the CompleteOrEscalate tool, the cancel and reason arguments will be populated dynamically by an LLM, similar to function tool arguments. A few lines below, there is the json_schema_extra variable that is used to provide some examples of how these arguments should be filled.


Primary assistant (routing) tools

In contrast to the Spending and the Recommendation assistants, no tool usage is implied for the Primary assistant, at least based on its system prompt defined previously. Nonetheless, and given the "routing" dimension of tools that we have just discovered, we can design a few tools for the Primary assistant also.

The code below implements the ToSpendingAssistant and ToRecommendationAssistant routing tools. 


from pydantic import Field

class ToSpendingAssistant(BaseModel):
    """Transfers work to a specialized assistant to handle billing inquiries."""

    request: str = Field(
        description="Any necessary followup questions the spending assistant should clarify before proceeding."
    )


class ToRecommendationAssistant(BaseModel):
    """Transfers work to a specialized assistant to handle electricity plan recommendations."""

    request: str = Field(
        description="Any necessary followup questions the recommendation assistant should clarify before proceeding."
    )


The two tools above will be triggered by the Primary assistant conditionally, depending on whether the flow of the conversation should be handled by the Spending or the Recommendation assistant. The request filled will be populated by an LLM dynamically.
