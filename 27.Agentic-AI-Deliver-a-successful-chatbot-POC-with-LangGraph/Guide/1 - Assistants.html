In this lecture we will implement our three assistants, namely the Primary, the Spending, and the Recommendation assistant.


Base Assistant class

We start with a base assistant class which will be inherited by all assistants. 


from langchain_core.runnables import Runnable, RunnableConfig
from langgraph.prebuilt import tools_condition
from langgraph.graph import END

class Assistant:
    """Base assistant class which will be inherited by other assistants."""

    def __init__(self, runnable: Runnable, name: str, tools: list):
        self.runnable = runnable
        self.name = name
        self.tools = tools

    def __call__(self, state: ChatbotState, config: RunnableConfig):
        while True:
            result = self.runnable.invoke(state)

            if not result.tool_calls and (
                not result.content
                or isinstance(result.content, list)
                and not result.content[0].get("text")
            ):
                messages = state["messages"] + [("user", "Respond with a real output.")]
                state = {**state, "messages": messages}
            else:
                break
        return {"messages": result}

    def route_non_primary_assistants(self, state: ChatbotState):
        route = tools_condition(state)
        if route == END:
            return END
        tool_calls = state["messages"][-1].tool_calls
        did_cancel = any(tc["name"] == CompleteOrEscalate.__name__ for tc in tool_calls)
        if did_cancel:
            return "leave_skill"
        return f"{self.name}_tools"


To initialize an instance of the Assistant class, we need to specify:

1) A Runnable, which in our case will be an LLM interface that implements the invoke method (example: ChatOpenAI or ChatOllama),

2) A string name of the assistant (example: "spending_asst")

3) A list of LangGraph tools, implemented either as functions or classes, as we discussed in the corresponding lecture (example: [fetch_spending_events, CompleteOrEscalate]).


The __call__() method basically returns only the meaningful output of the LLM that powers the assistant. That is, there will be conversation states where invoking the assistant's LLM will result in responses with empty content (e.g., when calling tools), and it makes no sense to communicate these to the end user. However, notice that the state is still updated to keep track of the changes in the dialog context. 


To better understand the route_non_primary_assistant, we first need to clarify two things about a LangGraph graph:

1) First, there is a special type of node called END node that indicates that a message is addressed to the user.

2) Second, tool sets in LangGraph are stored in dedicated nodes. In order for an assistant to access a tool, it has to interact with the corresponding tool node.

The figure below illustrates our graph based on the above remarks (focus on the Spending and the Recommendation assistants, i.e., the non-primary assistants of the graph for now):



Both Spending and Recommendation assistant have three conditional edges (dashed arrows), meaning that they can route to three nodes conditionally. These are:

1) the END node, meaning that the assistant will respond to the user,

2) the tool node (i.e., "Tools of Spending asst" and "Tools of Recommendation asst" respectively), meaning that the assistant decides to call one of its tools, and

3) the "leave_skill" node, which is a special node visited after the CompleteOrEscalate tool is called to escape from the current workflow. Notice that once the flow reaches the "leave_skill" node, then it can only go towards the Primary assistant. This is a behavior that we want, so that the Primary assistant can decide whether to end the conversation or re-route to a different assistant.

With these in mind, we can understand better the route_non_primary_assistants method now. First, we get the route variable from the tools_condition() function. The route variable will either be "__end__" or "tools", depending on whether the assistant deems that they should respond directly to the user or if they have to call a tool (and hence access their tool node). Also, from the last message stored in the ChatbotState we check if the CompleteOrEscalate tool has been called, to enable the assistant to route towards the "leave_skill" node.


With the above in mind, let's proceed to the implementation of the Spending and the Recommendation assistant.


Spending and Recommendation assistants


class SpendingAssistant(Assistant):
    """Spending/billing assistant class."""

    def __init__(self, llm: Runnable, name="spending_assistant"):
        tools = [fetch_spending_events, CompleteOrEscalate]
        runnable = spending_prompt | llm.bind_tools(tools)
        super().__init__(runnable=runnable, name=name, tools=tools)


class RecommendationAssistant(Assistant):
    """Plan recommendation assistant class."""

    def __init__(self, llm: Runnable, name="recommendation_assistant"):
        tools = [list_supported_plans, fetch_plan_description, CompleteOrEscalate]
        runnable = recommendation_prompt | llm.bind_tools(tools)
        super().__init__(runnable=runnable, name=name, tools=tools)


In the above, notice how we chain the corresponding system prompts of the assistants with the LLM runnable. Note that the latter is binded with the respective assistant tools. As such, each assistant is assigned with a tailored runnable.


Primary assistant

For the most part, the Primary assistant is similar to the other two assistants. However, note that the Primary assistant has different routing options compared to the second-level assistants. According to the figure above, the Primary assistant can:

1) respond to the user directly (END node),

2) route to the Spending or the Recommendation assistants, or

3) call any of its tools.

In fact, recall that we have implemented the ToSpendingAssistant and ToRecommendationAssistant routing tools for the Primary assistant to route between the two distinct workflows. Therefore, 2) and 3) coincide here.

The routing function of the Primary assistant is shown below: 


class PrimaryAssistant(Assistant):
    """Primary assistant class."""

    def __init__(self, llm: Runnable, name="primary_assistant"):
        tools = [ToSpendingAssistant, ToRecommendationAssistant]
        runnable = primary_prompt | llm.bind_tools(tools)
        super().__init__(runnable=runnable, name=name, tools=tools)

    def route_primary_assistant(self, state: ChatbotState):
        route = tools_condition(state)
        if route == END:
            return END
        tool_calls = state["messages"][-1].tool_calls
        if tool_calls:
            if tool_calls[0]["name"] == ToSpendingAssistant.__name__:
                return "enter_spending"
            elif tool_calls[0]["name"] == ToRecommendationAssistant.__name__:
                return "enter_recommendation"
            return "primary_assistant_tools"
        raise ValueError("Invalid route")


Notice that if the ToSpendingAssistant tool is called, the routing function will return the "enter_spending" node. Similarly, if the ToRecommendationAssistant tool is called, the routing function will return the "enter_recommendation" node.

But what are these "enter_*" nodes anyway? To answer this, let me show you how the actual LangGraph graph that we will be coding looks like: 



To put it simply, we will interpose an "enter_*" node between the Primary and each second-level assistant. These "enter_*" nodes will have a single purpose: to set the scope and role of each second-level assistant. Without the "enter_*" nodes, and given that each assistant has a view of the entire state and hence the history of messages (including prompts of other assistants, tool callings, Human and AI messages), the assistant might become confused of what its role is within this architecture. So the "enter" nodes always set the correct scope of assistants by communicating a role- and scope-setting message just before the second-level assistant takes control, ensuring that they understand their role and behave as expected.
